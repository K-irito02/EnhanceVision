# 任务控制模式详解

> **创建日期**: 2026-04-07  
> **适用版本**: EnhanceVision v1.x  
> **相关文件**: `ProcessingController.cpp`, `SettingsPage.qml`, `MessageItem.qml`

---

## 一、概述

EnhanceVision 提供三种任务控制模式，用于管理消息卡片的暂停/继续行为。用户可在 **设置 → 任务控制** 中选择适合的模式。

| 模式 | 名称 | 核心特点 |
|------|------|----------|
| 模式0 | 单任务暂停 | 暂停后继续处理其他消息 |
| 模式1 | 顺序暂停 | 暂停后阻塞整个队列（默认） |
| 模式2 | 自由选择 | 暂停后可手动选择处理哪些消息 |

---

## 二、模式详解

### 2.1 模式0：单任务暂停

#### 功能描述

当用户暂停某个消息卡片时：
- 该消息的所有任务从调度队列中被跳过
- 只有一个文件显示暂停图标（正在处理的文件，或第一个待处理文件）
- 其他消息卡片的任务继续正常处理
- 新上传的消息、手动重试的消息都会正常进入队列处理

#### 暂停行为

```
用户点击消息A的"暂停"按钮
    ↓
消息A添加到暂停集合 (m_pausedMessageIds)
    ↓
标记一个文件为暂停状态（UI显示暂停图标）
    ├── 优先：正在处理的文件
    └── 备选：第一个待处理文件
    ↓
释放正在处理任务的资源和AI引擎
    ↓
继续处理其他消息的任务
```

#### 恢复行为

```
用户点击消息A的"继续"按钮
    ↓
消息A从暂停集合中移除
    ↓
消息A添加到优先恢复队列 (m_priorityResumeMessageIds)
    ↓
检查当前是否有其他消息正在处理
    ├── 无其他消息处理中 → 立即开始处理消息A
    └── 有其他消息处理中 → 等待该消息的所有任务完成后优先处理消息A
        └── 注意：被暂停的任务不会立即恢复，而是等待当前消息完成
```

**重要说明**：
- 如果消息A暂停时有正在处理的文件（如视频），恢复后该文件不会立即继续处理
- 必须等待当前正在处理的消息B的**所有文件**都完成后，才会开始处理消息A
- 这确保了消息级别的完整性，避免多个消息的任务交叉执行

#### 适用场景

- 需要临时跳过某个大文件或耗时任务
- 希望其他任务不受影响继续处理
- 适合批量处理时的灵活控制

### 2.2 模式1：顺序暂停

#### 功能描述

当用户暂停某个消息卡片时：
- 该消息的任务被暂停
- 整个处理队列被阻塞
- 必须恢复该消息后，队列才能继续

#### 暂停行为

```
用户点击消息A的"暂停"按钮
    ↓
消息A添加到暂停集合
    ↓
标记正在处理的文件为暂停状态
    ├── 有正在处理的文件 → 标记该文件
    └── 无正在处理的文件 → 标记第一个待处理文件
    ↓
释放资源和AI引擎
    ↓
队列阻塞，不处理任何新任务
```

#### 恢复行为

```
用户点击消息A的"继续"按钮
    ↓
消息A从暂停集合中移除
    ↓
暂停的任务恢复为待处理状态
    ↓
队列解除阻塞，继续处理任务
```

#### 适用场景

- 需要严格按顺序处理任务
- 希望完全控制处理流程
- 适合需要人工干预的场景

---

### 2.3 模式2：自由选择

#### 功能描述

当用户暂停某个消息卡片时：
- 该消息被暂停
- 启用全局暂停标志
- 用户可以手动选择恢复哪些消息进行处理

#### 暂停行为

```
用户点击消息A的"暂停"按钮
    ↓
消息A添加到暂停集合
    ↓
启用全局暂停标志 (m_globalPauseEnabled = true)
    ↓
标记正在处理的文件为暂停状态
    ↓
释放资源和AI引擎
    ↓
队列暂停，等待用户选择
```

#### 恢复行为

```
用户点击消息A的"继续"按钮
    ↓
消息A从暂停集合中移除
    ↓
暂停的任务恢复为待处理状态
    ↓
检查是否还有其他暂停的消息
    ├── 有其他暂停消息 → 全局暂停保持
    └── 无其他暂停消息 → 解除全局暂停
    ↓
触发处理下一个任务
```

#### 适用场景

- 需要灵活选择处理哪些消息
- 多个消息需要不同的处理优先级
- 适合复杂的批量处理场景

## 三、操作步骤

### 3.1 切换任务控制模式

1. 打开应用程序
2. 点击左侧导航栏的 **设置** 图标
3. 找到 **任务控制** 区域
4. 选择所需的模式：
   - **单任务暂停**：暂停后继续处理其他消息
   - **顺序暂停**：暂停后阻塞整个队列（默认）
   - **自由选择**：暂停后可手动选择处理哪些消息

### 3.2 暂停消息任务

1. 在消息列表中找到目标消息卡片
2. 点击消息卡片右上角的 **暂停** 按钮（⏸️ 图标）
3. 观察变化：
   - 消息状态变为"已暂停"
   - 进度条保持当前位置
   - 根据模式不同，其他消息可能继续处理或被阻塞

### 3.3 恢复消息任务

1. 在消息列表中找到已暂停的消息卡片
2. 点击消息卡片右上角的 **继续** 按钮（▶️ 图标）
3. 观察变化：
   - 消息状态恢复为"处理中"或"等待处理"
   - 任务重新进入处理队列

## 四、技术实现

### 4.1 核心数据结构

```cpp
// 暂停的消息ID集合
QSet<QString> m_pausedMessageIds;

// 全局暂停标志（模式2使用）
bool m_globalPauseEnabled = false;

// 优先恢复队列（模式0使用）
QList<QString> m_priorityResumeMessageIds;

// 当前正在处理的消息ID（模式0使用，用于跟踪消息完成状态）
QString m_currentProcessingMessageId;
```

### 4.2 关键函数

| 函数 | 功能 |
|------|------|
| `pauseMessageTasks()` | 暂停指定消息的任务 |
| `resumeMessageTasks()` | 恢复指定消息的任务 |
| `processNextTask()` | 调度下一个待处理任务 |
| `isMessagePaused()` | 检查消息是否被暂停 |
| `clearAllPauseStates()` | 清除所有暂停状态 |

### 4.3 调度逻辑

```
processNextTask() 调用
    ↓
检查队列状态和处理计数
    ↓
根据暂停模式检查阻塞条件
    ├── 模式1：队首消息暂停则阻塞
    └── 模式2：全局暂停则阻塞
    ↓
模式0：检查当前消息完成状态
    ├── 当前消息还有待处理任务 → 继续处理当前消息
    └── 当前消息已完成 → 检查优先恢复队列
        ├── 有优先消息 → 只处理该消息的任务
        └── 无优先消息 → 正常FIFO调度
    ↓
跳过暂停消息的任务
    ↓
检查AI引擎可用性
    ↓
启动任务处理，更新 m_currentProcessingMessageId
```

## 五、模式对比

| 特性 | 模式0 | 模式1 | 模式2 |
|------|-------|-------|-------|
| 暂停后队列行为 | 继续处理其他消息 | 阻塞整个队列 | 阻塞整个队列 |
| 暂停图标显示 | 一个文件 | 一个文件 | 正在处理的文件 |
| 恢复优先级 | 等当前消息**所有任务**完成后优先 | 立即恢复 | 立即恢复 |
| 全局暂停标志 | 不使用 | 不使用 | 使用 |
| 删除暂停消息 | 不传递暂停状态 | 传递给下一消息 | 传递给下一消息 |
| 适用场景 | 灵活跳过 | 严格顺序 | 手动选择 |

## 六、常见问题

### Q1: 暂停后为什么只有一个文件显示暂停图标？

**A**: 这是设计行为。暂停图标用于指示当前被暂停的处理点，而不是所有待处理的文件。该消息的所有任务都会被逻辑暂停（从调度中跳过），但只有一个文件在UI上显示暂停状态。

### Q2: 模式0下恢复消息后，为什么不是立即处理？

**A**: 如果当前有其他消息正在处理，恢复的消息会被添加到优先恢复队列。等当前消息的**所有任务**完成后，会优先处理恢复的消息，而不是按原来的队列顺序。这确保了消息级别的完整性，避免多个消息的任务交叉执行。

### Q3: 模式0下删除暂停消息的所有文件会发生什么？

**A**: 当删除暂停消息的所有文件后：
- 该消息会从暂停集合和优先恢复队列中移除
- **不会**将暂停状态传递给其他消息
- 当前正在处理的消息会继续正常处理
- 这与模式1和模式2不同，后者会将暂停状态传递给下一个消息

### Q4: 如何清除所有暂停状态？

**A**: 可以通过以下方式：
1. 逐个点击暂停消息的"继续"按钮
2. 切换会话（会自动清理当前会话的暂停状态）
3. 重启应用程序
