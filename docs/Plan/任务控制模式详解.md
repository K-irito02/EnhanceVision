# 任务控制模式详解

> **创建日期**: 2026-04-07  
> **适用版本**: EnhanceVision v1.x  
> **相关文件**: `ProcessingController.cpp`, `SettingsPage.qml`, `MessageItem.qml`

---

## 一、概述

EnhanceVision 提供三种任务控制模式，用于管理消息卡片的暂停/继续行为。用户可在 **设置 → 任务控制** 中选择适合的模式。

| 模式 | 名称 | 核心特点 |
|------|------|----------|
| 模式0 | 单任务暂停 | 暂停后继续处理其他消息 |
| 模式1 | 顺序暂停 | 暂停后阻塞整个队列（默认） |
| 模式2 | 自由选择 | 暂停后可手动选择处理哪些消息 |

---

## 1.1 启动恢复（待恢复）

当应用关闭（正常关闭或异常退出）时，如果仍存在未完成的消息卡片（`处理中 / 等待中 / 暂停中`），下次启动会进入统一的启动恢复流程：

- 未完成的消息卡片先统一显示为 `待恢复`（只用于启动恢复阶段的临时状态）
- 启动时弹出恢复对话框，用户二选一：
  - `恢复关闭前状态`：按关闭前的队列语义与暂停语义重建队列并继续
  - `全部标记为失败`：将这些未完成任务统一标记为失败，允许用户后续手动重新处理
- 在恢复决策完成前，不允许切换任务控制模式
- 恢复决策完成后，只要仍存在 `待恢复 / 处理中 / 等待中 / 暂停中` 的消息卡片，也不允许切换任务控制模式

恢复相关实现入口：
- `TaskRecoveryController`：负责加载/保存快照、生成摘要、触发恢复/失败决策
- `ProcessingController::buildRecoverySnapshot()` / `ProcessingController::restoreFromRecoverySnapshot()`：构建与恢复队列语义

## 二、模式详解

### 2.1 模式0：单任务暂停

#### 功能描述

当用户暂停某个消息卡片时：
- 该消息的所有任务从调度队列中被跳过
- 只有一个文件显示暂停图标（正在处理的文件，或第一个待处理文件）
- 其他消息卡片的任务继续正常处理
- 新上传的消息、手动重试的消息都会正常进入队列处理

#### 暂停行为

```
用户点击消息A的"暂停"按钮
    ↓
消息A添加到暂停集合 (m_pausedMessageIds)
    ↓
标记一个文件为暂停状态（UI显示暂停图标）
    ├── 优先：正在处理的文件
    └── 备选：第一个待处理文件
    ↓
释放正在处理任务的资源和AI引擎
    ↓
继续处理其他消息的任务
```

#### 恢复行为

```
用户点击消息A的"继续"按钮
    ↓
消息A从暂停集合中移除
    ↓
消息A添加到优先恢复队列 (m_priorityResumeMessageIds)
    ↓
检查当前是否有其他消息正在处理
    ├── 无其他消息处理中 → 立即开始处理消息A
    └── 有其他消息处理中 → 等待该消息的所有任务完成后优先处理消息A
        └── 注意：被暂停的任务不会立即恢复，而是等待当前消息完成
```

**重要说明**：
- 如果消息A暂停时有正在处理的文件（如视频），恢复后该文件不会立即继续处理
- 必须等待当前正在处理的消息B的**所有文件**都完成后，才会开始处理消息A
- 这确保了消息级别的完整性，避免多个消息的任务交叉执行

#### 适用场景

- 需要临时跳过某个大文件或耗时任务
- 希望其他任务不受影响继续处理
- 适合批量处理时的灵活控制

---

### 2.2 模式1：顺序暂停

#### 功能描述

当用户暂停某个消息卡片时：
- 该消息的任务被暂停
- 整个处理队列被阻塞
- 必须恢复该消息后，队列才能继续

#### 暂停行为

```
用户点击消息A的"暂停"按钮
    ↓
消息A添加到暂停集合
    ↓
标记正在处理的文件为暂停状态
    ├── 有正在处理的文件 → 标记该文件
    └── 无正在处理的文件 → 标记第一个待处理文件
    ↓
释放资源和AI引擎
    ↓
队列阻塞，不处理任何新任务
```

#### 恢复行为

```
用户点击消息A的"继续"按钮
    ↓
消息A从暂停集合中移除
    ↓
暂停的任务恢复为待处理状态
    ↓
队列解除阻塞，继续处理任务
```

#### 适用场景

- 需要严格按顺序处理任务
- 希望完全控制处理流程
- 适合需要人工干预的场景

---

### 2.3 模式2：自由选择（选择性启动模式）

#### 功能描述

此模式的核心特点是**默认暂停**，用户需要手动选择启动哪些消息的处理：
- 新上传的消息卡片默认为暂停状态，不会自动开始处理
- 重新处理的消息也默认为暂停状态
- 只有用户点击"继续"按钮后，消息才会开始处理
- 暂停后可以继续处理其他已激活的消息
- 恢复时若有其他消息正在处理，需等待其完成

#### 上传行为

```
用户上传多媒体文件并发送
    ↓
消息添加到暂停集合 (m_pausedMessageIds)
    ↓
所有任务标记为暂停状态 (Paused)
    ↓
消息卡片显示暂停图标
    ↓
队列不会自动处理该消息（等待用户点击"继续"）
```

#### 暂停行为（对正在处理的消息）

```
用户点击消息A的"暂停"按钮
    ↓
消息A添加到暂停集合
    ↓
消息A从激活集合中移除 (m_activatedMessageIds)
    ↓
标记正在处理的文件为暂停状态
    ↓
释放资源和AI引擎
    ↓
继续处理其他已激活的消息
```

#### 恢复行为

```
用户点击消息A的"继续"按钮
    ↓
消息A从暂停集合中移除
    ↓
消息A添加到激活集合和优先恢复队列（队列末尾）
    ↓
检查当前是否有其他消息正在处理
    ├── 无其他消息处理中 → 立即开始处理消息A
    └── 有消息B正在处理中 → 等待B的所有文件完成后按队列顺序处理
```

#### 处理顺序保证

```
用户依次激活消息A、B、C
    ↓
优先恢复队列：[A, B, C]（按激活顺序）
    ↓
处理顺序：A → B → C（严格按激活顺序）
```

**重要说明**：
- 所有激活的消息都按激活顺序添加到优先恢复队列末尾
- 处理顺序严格按照激活顺序执行
- 暂停后恢复的消息也添加到队列末尾，不会插队

#### 再次暂停后恢复

```
消息A正在处理中，用户点击"暂停"
    ↓
消息A暂停，继续处理后续消息B
    ↓
用户点击消息A的"继续"
    ├── B正在处理中 → 等待B完成后再处理A
    └── B已完成 → 立即处理A
```

#### 适用场景

- 需要完全控制处理顺序
- 希望手动选择处理哪些消息
- 适合需要根据优先级选择性处理的场景
- 适合不希望自动开始处理的场景

## 三、操作步骤

### 3.1 切换任务控制模式

1. 打开应用程序
2. 点击左侧导航栏的 **设置** 图标
3. 找到 **任务控制** 区域
4. 选择所需的模式：
   - **单任务暂停**：暂停后继续处理其他消息
   - **顺序暂停**：暂停后阻塞整个队列（默认）
   - **自由选择**：暂停后可手动选择处理哪些消息

### 3.2 暂停消息任务

1. 在消息列表中找到目标消息卡片
2. 点击消息卡片右上角的 **暂停** 按钮（⏸️ 图标）
3. 观察变化：
   - 消息状态变为"已暂停"
   - 进度条保持当前位置
   - 根据模式不同，其他消息可能继续处理或被阻塞

### 3.3 恢复消息任务

1. 在消息列表中找到已暂停的消息卡片
2. 点击消息卡片右上角的 **继续** 按钮（▶️ 图标）
3. 观察变化：
   - 消息状态恢复为"处理中"或"等待处理"
   - 任务重新进入处理队列

## 四、技术实现

### 4.1 核心数据结构

```cpp
// 暂停的消息ID集合（所有模式使用）
QSet<QString> m_pausedMessageIds;

// 已激活的消息ID集合（模式2使用，用户点击过"继续"的消息）
QSet<QString> m_activatedMessageIds;

// 优先恢复队列（模式0和模式2使用）
QList<QString> m_priorityResumeMessageIds;

// 当前正在处理的消息ID（模式0和模式2使用，用于跟踪消息完成状态）
QString m_currentProcessingMessageId;

// 全局暂停标志（已废弃，保留兼容）
bool m_globalPauseEnabled = false;
```

启动恢复阶段临时状态：

```cpp
// DataTypes.h
enum class ProcessingStatus {
    Pending,
    Processing,
    Completed,
    Failed,
    Cancelled,
    Paused,
    Recoverable, // 待恢复：仅用于启动恢复阶段
};
```

### 4.2 关键函数

| 函数 | 功能 |
|------|------|
| `pauseMessageTasks()` | 暂停指定消息的任务 |
| `resumeMessageTasks()` | 恢复指定消息的任务 |
| `processNextTask()` | 调度下一个待处理任务 |
| `isMessagePaused()` | 检查消息是否被暂停 |
| `clearAllPauseStates()` | 清除所有暂停状态 |

### 4.3 调度逻辑

```
processNextTask() 调用
    ↓
检查队列状态和处理计数
    ↓
根据暂停模式检查阻塞条件
    ├── 模式1：队首消息暂停则阻塞
    └── 模式2：无已激活消息则不处理
    ↓
模式0和模式2：检查当前消息完成状态
    ├── 当前消息还有待处理任务 → 继续处理当前消息
    └── 当前消息已完成 → 检查优先恢复队列
        ├── 有优先消息 → 只处理该消息的任务
        └── 无优先消息 → 模式0正常FIFO，模式2只处理已激活消息
    ↓
跳过暂停消息的任务
    ↓
模式2：跳过未激活消息的任务
    ↓
检查AI引擎可用性
    ↓
启动任务处理，更新 m_currentProcessingMessageId
```

## 五、模式对比

| 特性 | 模式0 | 模式1 | 模式2 |
|------|-------|-------|-------|
| **上传后默认状态** | 自动处理 | 自动处理 | **暂停状态** |
| **暂停后队列行为** | 继续处理其他消息 | 阻塞整个队列 | 继续处理其他已激活消息 |
| **暂停图标显示** | 一个文件 | 一个文件 | 一个文件 |
| **恢复优先级** | 等当前消息完成后优先 | 立即恢复 | 等当前消息完成后优先 |
| **优先恢复队列** | 使用 | 不使用 | 使用 |
| **已激活消息集合** | 不使用 | 不使用 | 使用 |
| **删除暂停消息** | 不传递暂停状态 | 传递给下一消息 | 不传递暂停状态 |
| **适用场景** | 灵活跳过 | 严格顺序 | 选择性启动 |

## 六、常见问题

### Q1: 暂停后为什么只有一个文件显示暂停图标？

**A**: 这是设计行为。暂停图标用于指示当前被暂停的处理点，而不是所有待处理的文件。该消息的所有任务都会被逻辑暂停（从调度中跳过），但只有一个文件在UI上显示暂停状态。

### Q2: 模式0下恢复消息后，为什么不是立即处理？

**A**: 如果当前有其他消息正在处理，恢复的消息会被添加到优先恢复队列。等当前消息的**所有任务**完成后，会优先处理恢复的消息，而不是按原来的队列顺序。这确保了消息级别的完整性，避免多个消息的任务交叉执行。

### Q3: 模式0下删除暂停消息的所有文件会发生什么？

**A**: 当删除暂停消息的所有文件后：
- 该消息会从暂停集合和优先恢复队列中移除
- **不会**将暂停状态传递给其他消息
- 当前正在处理的消息会继续正常处理
- 这与模式1不同，模式1会将暂停状态传递给下一个消息

### Q5: 模式2下为什么上传的消息不会自动开始处理？

**A**: 这是模式2（自由选择模式）的核心设计。在此模式下：
- 所有新上传的消息默认为暂停状态
- 用户需要手动点击"继续"按钮来启动处理
- 这允许用户完全控制处理哪些消息以及处理顺序

### Q6: 模式2下暂停正在处理的消息后会发生什么？

**A**: 暂停正在处理的消息A后：
- 消息A的处理会暂停，资源被释放
- 其他已激活的消息（用户之前点击过"继续"的消息）会继续处理
- 如果没有其他已激活的消息，队列会进入空闲状态
- 当用户再次点击消息A的"继续"时，如果有其他消息正在处理，需要等待其完成

### Q4: 如何清除所有暂停状态？

**A**: 可以通过以下方式：
1. 逐个点击暂停消息的"继续"按钮
2. 切换会话（会自动清理当前会话的暂停状态）
3. 重启应用程序
