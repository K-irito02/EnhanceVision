## 0. 结论先行（最核心问题判断）

### 最根本问题不是“线程数量不够”，而是：
1. **UI 主线程上的高频数据变更 + 大粒度重绘链路**（首要根因）
2. **后端状态同步路径过于频繁且遍历粒度偏粗**（第二根因）
3. 多线程设计本身已有改进（异步化、限流、优先级让渡），**不是当前第一瓶颈**

> 结论归类：
> - 多线程设计：**有优化空间，但不是当前卡顿第一原因**
> - QML 渲染/绑定性能：**是第一原因**
> - 后端逻辑（状态同步与查找复杂度）：**是第二原因，放大第一原因**

## 1. 证据链（从代码现状出发）

## 1.1 线程侧已做了关键正确动作（说明不是首因）
- `ProcessingController` 已将线程池优先级降为 `QThread::LowPriority`。
- 视频缩略图 + Shader 处理走 `processShaderVideoThumbnailAsync(...)` 异步链路。
- 有会话切换时优先级让渡（`onSessionChanging(...)`）。
- 有并发削峰（`setImportBurstMode(...)` / `refreshEffectiveConcurrencyLimit(...)`）。

这些改动说明“纯粹因为重计算都在主线程”这个老问题已被部分解决。

## 1.2 当前真正高压点：QML 列表 delegate 重建过于频繁
- `MessageModel::updateFileStatus(...)` 每次文件状态变化都 `emit dataChanged(..., {MediaFilesRole})`。
- `MessageList.qml` 的 `delegate` 中监听 `onMediaFilesChanged` 后会执行 `_buildMediaForDelegate()`。
- `_buildMediaForDelegate()` 每次会遍历整个 `model.mediaFiles`，并对 `_cachedMedia` 执行多次 `set/append/remove`。
- `MessageItem.qml` 里 `_updateFileCounts()` 也会对每条消息的全部文件再次遍历统计。

结果：处理中高频状态更新时，形成“**C++ dataChanged -> QML rebuild listmodel -> JS 全量遍历**”链路。

## 1.3 后端状态同步频率高，放大 UI 抖动
- `updateTaskProgress(...)` 虽有节流，但仍会触发 `syncMessageProgress(...)` 与 `syncModelTask(...)`。
- `completeTask(...)` / `failTask(...)` / retry 路径多次触发：
  - `syncMessageProgress(...)`
  - `syncMessageStatus(...)`
  - `requestSessionSync()`
- `requestSessionSync()` 是 500ms debounce，但处理密集时持续被触发。

这会导致 UI 在交互（拖拽窗口、切会话、滑动卡片、点按钮）期间仍持续收到大量模型更新，主线程事件循环被占用。

## 1.4 会话切换卡顿的直接触发点
- `MessageModel::loadFromSession(...)` 使用 `beginResetModel()/endResetModel()` 全量重置。
- `MessageModel::clear()` 也会 reset。

会话切换时这类 reset 会触发列表整页重建，如果恰逢后台任务高频更新，卡顿体感明显。

## 2. 核心治理策略（按优先级）

## P0（必须先做）：切断“高频全量重建”链路

### P0-1. 将 `MediaFilesRole` 全量更新改为“文件级增量更新”
**目标**：单文件状态变更不再触发整条消息媒体列表重建。

实施思路：
1. 在 `MessageModel` 新增“文件级变更信号”（例如 `mediaFileUpdated(messageId, fileId, changedFields)`）。
2. `MessageList` delegate 不再在 `onMediaFilesChanged` 中全量 `_buildMediaForDelegate()`。
3. 改为只定位 `_cachedMedia` 对应 file 行，执行单行 `setProperty`。

验收标准：
- 处理中 `onMediaFilesChanged` 触发次数显著下降。
- 单次文件完成仅更新 1 个缩略项，不重扫全部文件。

### P0-2. 为 `MessageItem` 的文件统计加节流与缓存
**目标**：避免每次 `dataChanged` 都 O(N) 扫描所有文件。

实施思路：
1. 将 `_updateFileCounts()` 改为 66~100ms 的 timer 合并执行。
2. 对文件状态变化做增量计数（+1/-1），避免全量重算。

验收标准：
- `MessageItem` 内部统计函数调用频率下降到每秒 <= 10~15 次/消息。

### P0-3. 会话切换避免 `beginResetModel` 大锤
**目标**：降低切会话时主线程峰值开销。

实施思路：
1. `loadFromSession(...)` 从 reset 改为差量更新（最少化 remove/insert/dataChanged）。
2. 若短期内无法完全差量化，至少在切换阶段启用“UI 冻结窗口”：
   - 暂停处理进度回传到可见模型（只更新会话内存态）。
   - 切换完成后一次性合并可见更新。

验收标准：
- 会话切换耗时下降，切换时不再出现明显掉帧/输入停顿。

## P1（紧随其后）：后端同步路径瘦身

### P1-1. 降低 `syncMessageProgress/syncMessageStatus` 调用频率
实施思路：
1. 在 `ProcessingController` 增加“按 messageId 的统一节流器”（例如 100ms）。
2. 进度/状态合并后一次下发，避免同一 tick 多次同步。

### P1-2. 降低 session 查找复杂度
现状有 `sessionIdForMessage(...)`、`messageInSession(...)` 的遍历查找。

实施思路：
1. 在 `SessionController` 维护 `messageId -> sessionId` 索引哈希。
2. 维护 `sessionId + messageId -> messageIndex` 索引。
3. 会话增删改时同步更新索引。

目标：把高频路径从 O(会话数×消息数) 降到近 O(1)。

### P1-3. `requestSessionSync` 分层
实施思路：
1. 区分“仅内存同步”与“持久化写回”。
2. 处理中仅做内存态聚合，落盘进一步延后（如 1~2s，空闲时触发）。

## P2（验证与守护）：建立可量化基线

### P2-1. 埋点补齐
- `MessageList`：统计 `_buildMediaForDelegate` 调用次数/耗时。
- `MessageItem`：统计 `_updateFileCounts` 调用次数/耗时。
- `SessionController`：统计切会话总耗时。
- 帧统计：95/99 分位帧耗时（渲染线程与 GUI 线程）。

### P2-2. 回归场景（必须覆盖）
1. 处理中拖拽主窗口 30 秒。
2. 处理中切换会话来回 20 次。
3. 处理中快速滑动消息卡片列表。
4. 处理中连续点击主要按钮（重试/删除/下载/切换）。
5. 处理中批量上传多媒体（10/30/50 文件）。

### P2-3. 量化门槛
- 拖拽与滚动场景无连续 >100ms 卡顿。
- P95 帧时长 < 16.7ms，P99 < 24ms（Release）。
- 会话切换耗时下降到稳定阈值（目标 <120ms，视数据量可分档）。

## 5. 最终回答（针对提问）

你问“多线程设计有问题？QML 渲染优化有问题？还是后端逻辑有问题？”

**最终判断：**
- **第一核心问题：QML 渲染/绑定更新策略（高频+大粒度）**
- **第二核心问题：后端同步逻辑粒度与调用频率偏高**
- **多线程不是当前最核心问题**（已有关键异步化与并发治理基础）

因此本计划优先从 **UI 更新链路瘦身（P0）** 下手，再做 **后端同步路径优化（P1）**，这是最快、最稳、最符合当前代码现状的根因治理路径。
