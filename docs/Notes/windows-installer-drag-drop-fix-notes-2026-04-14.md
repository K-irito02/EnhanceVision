# Windows 安装版拖拽导入修复说明

## 概述

修复安装版 EnhanceVision 在 Windows 上运行后无法从资源管理器拖拽导入多媒体文件的问题。

**创建日期**: 2026-04-14

---

## 变更概述

### 修改文件

| 文件路径 | 修改内容 |
|----------|----------|
| `src/app/Application.cpp` | 在主窗口原生句柄创建后，按窗口级别放行 Windows 拖拽消息并启用原生文件拖放 |
| `include/EnhanceVision/app/Application.h` | 新增主窗口拖拽兼容初始化接口 |
| `CHANGELOG.md` | 记录安装版拖拽导入修复 |

---

## 问题现象

- 开发环境运行时可正常拖拽文件到应用
- 打包、安装后启动应用，拖拽图片或视频到文件列表区域无反应
- `qml/components/FileList.qml` 中的 `DropArea` 本身逻辑正常，问题主要出现在 Windows 安全边界与原生窗口消息放行阶段

---

## 原因分析

已有实现只在进程启动早期调用 `ChangeWindowMessageFilter`，存在两个问题：

1. 调用时机过早，主窗口句柄尚未创建
2. 旧接口偏进程级兼容处理，在安装版/UAC 场景下稳定性不足

安装版更容易落入“安装器提权 + 首启拖拽来源为普通权限 Explorer”这一组合，导致 `DropArea` 收不到底层拖放消息。

---

## 实现方案

在 `Application::initialize()` 中，拿到 `QQuickWindow` 后立即：

1. 通过 `winId()` 强制确保主窗口原生句柄已创建
2. 使用 `ChangeWindowMessageFilterEx(hwnd, ...)` 按窗口级别放行：
   - `WM_DROPFILES`
   - `WM_COPYDATA`
   - `0x0049` (`WM_COPYGLOBALDATA`)
3. 调用 `DragAcceptFiles(hwnd, TRUE)` 明确启用原生文件拖放

这样既保留了启动早期的兼容放行，也补齐了真正生效的窗口级配置。

## 后续建议

- 安装包验收时保留“完成页启动应用后拖拽文件”这一固定回归项
- 如果后续增加提权辅助工具或外部启动器，也要同步验证 Explorer 到主窗口的拖放链路
