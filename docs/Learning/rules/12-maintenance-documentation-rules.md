# 12 维护与文档同步规范

## 变更同步

- 功能/修复完成后同步更新 `docs/Notes/`
- 架构或流程变化需更新对应规则文件
- 用户可见行为变化需更新使用说明
- 涉及持久化、图标、日志、启动行为的改动时，README、规则与笔记必须一起更新，避免知识只留在代码里
- 涉及窗口宿主、缩放行为、日志收敛的改动时，必须同步记录根因、验证结果和保留的取舍
- 关闭链路、运行日志与媒体源加载守卫这类容易回归的改动，必须在笔记里写清楚：修复点、验证方式、仍然保留的外部环境告警

## 技术债管理

- 每次迭代清理可见技术债（重复、废弃、死代码）
- 临时代码必须有移除计划
- 严禁“先堆功能后治理”的长期拖延

## 评审检查点

- 是否破坏分层与边界
- 是否新增重复实现
- 是否影响性能基线
- 是否补充了必要文档

## 长期目标

- 代码可读、职责单一、修改可控
- 目录稳定、命名统一、知识可沉淀
