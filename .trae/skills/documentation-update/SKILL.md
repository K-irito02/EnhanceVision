# 文档更新工作流

完成功能开发或问题修复后，自动更新项目文档的工作流程。

## 触发条件

- 完成新功能开发
- 修复重要 Bug
- 重构代码结构
- 添加新组件或模块
- 用户明确要求更新文档

## 执行步骤

### 第一步：创建/更新笔记文件

在 `docs/Notes/` 目录下创建或更新笔记文件，记录：

1. **功能概述**：完成的功能描述
2. **新增文件**：新增的组件、类、文件列表
3. **修改文件**：修改的文件列表及修改内容
4. **遇到的问题**：开发过程中遇到的问题
5. **解决方案**：问题的解决方法和思路
6. **技术细节**：关键实现细节和代码片段
7. **参考资料**：参考的文档或资料链接

**文件命名规范**：
- 功能开发：`<feature-name>-implementation-notes.md`
- 问题修复：`<issue-name>-fix-notes.md`
- 重构：`<refactor-name>-notes.md`

**文件模板**：

```markdown
# <标题>

## 概述

<功能或问题概述>

**创建日期**: YYYY-MM-DD
**作者**: AI Assistant

---

## 一、完成的功能

### 1. 创建的新组件

| 组件 | 文件路径 | 功能描述 |
|------|---------|---------|
| ... | ... | ... |

### 2. 修改的文件

| 文件 | 修改内容 |
|------|---------|
| ... | ... |

### 3. 实现的功能特性

- ✅ 功能点 1
- ✅ 功能点 2

---

## 二、遇到的问题及解决方案

### 问题 1：<问题描述>

**现象**：...

**原因**：...

**解决方案**：...

---

## 三、技术实现细节

...

---

## 四、参考资料

- [链接标题](URL)
```

### 第二步：检查规则文件

检查 `.trae/rules/` 目录下的规则文件是否需要更新：

| 规则文件 | 检查内容 |
|---------|---------|
| `01-project-overview.md` | 新增的技术栈、目录结构变化 |
| `02-architecture.md` | 架构变化、新增模块 |
| `03-cpp-standards.md` | 新的 C++ 编码规范、模式 |
| `04-qml-standards.md` | 新的 QML 编码规范、模式 |
| `05-build-system.md` | 构建系统变化、新增依赖 |
| `06-i18n.md` | 国际化相关变化 |
| `07-performance.md` | 性能优化规则变化 |
| `08-ui-design.md` | UI 设计规范变化 |
| `09-shader-standards.md` | Shader 规范变化 |

**更新原则**：
- 只添加必要的内容，避免冗余
- 保持规则文件的简洁性和可读性
- 新增规则应该有明确的示例代码

### 第三步：检查 README 文件

检查以下 README 文件是否需要更新：

| 文件 | 检查内容 |
|------|---------|
| `README.md` | 项目概述、技术栈、目录结构、核心特性 |
| `QML-README.md` | QML 组件列表、目录结构、功能说明 |

**更新内容**：
- 新增的组件或模块
- 目录结构变化
- 技术栈更新
- 新增的核心特性

## 输出清单

完成文档更新后，确认以下内容：

- [ ] 笔记文件已创建/更新
- [ ] 规则文件已检查并更新（如需要）
- [ ] README 文件已检查并更新（如需要）
- [ ] 所有文档使用正确的语言（中文）

## 注意事项

1. **语言一致性**：所有文档使用中文编写
2. **代码示例**：使用正确的语法高亮标记
3. **文件路径**：使用相对路径或绝对路径
4. **日期格式**：使用 YYYY-MM-DD 格式
5. **表格格式**：使用标准 Markdown 表格语法
