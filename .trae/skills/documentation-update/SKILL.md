# 文档更新工作流

完成功能开发或问题修复后，系统化更新项目文档的工作流程。

## 核心原则

1. **及时更新** - 完成变更后立即更新文档
2. **保持一致** - 文档内容与代码实现同步
3. **简洁明了** - 避免冗余，突出关键信息
4. **结构清晰** - 遵循统一的文档格式

## 触发条件

| 场景 | 优先级 | 必须更新的文档 |
|------|--------|----------------|
| 新功能开发 | 高 | 笔记、README、规则文件 |
| Bug 修复 | 中 | 笔记、变更日志 |
| 代码重构 | 中 | 笔记、规则文件 |
| 新增组件/模块 | 高 | 笔记、README、API 文档 |
| 架构变更 | 高 | 笔记、规则文件、README |
| 依赖更新 | 中 | 笔记、构建文档 |
| 性能优化 | 中 | 笔记、性能文档 |

### 第二步：更新变更日志

检查并更新 `CHANGELOG.md`（如存在）或 `docs/CHANGELOG.md`。

#### 变更日志格式

```markdown
# Changelog

All notable changes to this project will be documented in this file.

## [Unreleased]

### Added
- 新增功能描述

### Changed
- 变更描述

### Fixed
- 修复描述

### Removed
- 移除描述

## [1.0.0] - 2024-01-15

### Added
- 初始版本功能列表
```

#### 变更类型分类

| 类型 | 说明 | 示例 |
|------|------|------|
| `Added` | 新增功能 | 新增暗色主题支持 |
| `Changed` | 功能变更 | 优化图像处理性能 |
| `Fixed` | Bug 修复 | 修复内存泄漏问题 |
| `Removed` | 移除功能 | 移除废弃的 API |
| `Deprecated` | 即将废弃 | 标记旧接口为废弃 |
| `Security` | 安全相关 | 修复安全漏洞 |

### 第三步：检查规则文件

检查 `.trae/rules/` 目录下的规则文件是否需要更新。

#### 规则文件检查清单

| 规则文件 | 检查内容 | 更新条件 |
|----------|----------|----------|
| `01-project-overview.md` | 技术栈、目录结构 | 新增技术或目录 |
| `02-architecture.md` | 架构设计、模块关系 | 架构变更 |
| `03-cpp-standards.md` | C++ 编码规范 | 新增编码模式 |
| `04-qml-standards.md` | QML 编码规范 | 新增组件模式 |
| `05-build-system.md` | 构建配置、依赖 | 构建变更 |
| `06-i18n.md` | 国际化规则 | 新增翻译 |
| `07-performance.md` | 性能优化规则 | 性能相关变更 |
| `08-ui-design.md` | UI 设计规范 | UI 变更 |
| `09-shader-standards.md` | Shader 规范 | Shader 变更 |
| `10-skills-and-mcp.md` | 技能和 MCP 配置 | 新增技能 |

#### 更新原则

1. **只添加必要内容** - 避免冗余
2. **保持简洁** - 规则文件应易于阅读
3. **突出变更** - 新增内容应有明确标记
4. **考虑复用** - 规则应具有通用性

### 第四步：更新 README 文件

检查并更新项目 README 文件。

#### README 检查清单

| 检查项 | 说明 |

### 第五步：更新 API 文档（如适用）

如果新增或修改了公共 API，需要更新 API 文档。

#### API 文档位置

- 头文件中的 Doxygen 注释
- `docs/api/` 目录下的文档

#### API 文档格式

```cpp
/**
 * @brief 简要描述
 *
 * 详细描述
 *
 * @param paramName 参数描述
 * @return 返回值描述
 *
 * @example
 * @code
 * auto result = functionName(param);
 * @endcode
 *
 * @note 注意事项
 * @warning 警告信息
 * @see 相关函数
 */
```

### 第六步：更新国际化文件（如适用）

如果新增或修改了用户可见的文本，需要更新翻译文件。

#### 翻译文件位置

```
resources/i18n/
├── app_zh_CN.ts    # 中文
└── app_en_US.ts    # 英文
```

#### 更新步骤

```powershell
# 1. 更新翻译源文件
lupdate CMakeLists.txt -ts resources/i18n/app_zh_CN.ts
lupdate CMakeLists.txt -ts resources/i18n/app_en_US.ts

# 2. 使用 Qt Linguist 编辑翻译
linguist resources/i18n/app_zh_CN.ts

# 3. 生成翻译二进制文件
lrelease resources/i18n/app_zh_CN.ts
lrelease resources/i18n/app_en_US.ts
```

## 最佳实践

### 文档编写原则

1. **面向读者** - 考虑谁会阅读这份文档
2. **结构清晰** - 使用标题和列表组织内容
3. **示例丰富** - 提供可运行的代码示例
4. **及时更新** - 代码变更后立即更新文档

### 常见错误避免

1. **过时文档** - 定期检查和更新
2. **冗余内容** - 删除重复和过时信息
3. **缺失示例** - 添加必要的代码示例
4. **链接失效** - 定期检查外部链接

### 文档维护建议

1. 每次代码变更后检查相关文档
2. 定期进行文档审查（每月/每季度）
3. 收集用户反馈改进文档
4. 保持文档版本与代码版本同步
