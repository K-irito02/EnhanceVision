## 二、技术方案审视

### 2.1 架构方案：Qt WebEngine + React + QWebChannel

#### 当前方案
```
┌─────────────────────────────────────────────────────────────┐
│                    EnhanceVision.exe (单个进程)              │
│  ┌───────────────────────────────────────────────────────┐  │
│  │              Qt WebEngine (Chromium 108)               │  │
│  │  ┌─────────────────────────────────────────────────┐  │  │
│  │  │           React 前端                              │  │  │
│  │  └─────────────────────────────────────────────────┘  │  │
│  │                     ↕ QWebChannel                     │  │  │
│  │  ┌─────────────────────────────────────────────────┐  │  │
│  │  │           C++ 后端 (NCNN + Shader + FFmpeg)      │  │  │
│  │  └─────────────────────────────────────────────────┘  │  │
│  └───────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

#### ⚠️ 问题分析

| 问题 | 严重程度 | 说明 |
|------|---------|------|
| **性能开销大** | 🔴 高 | WebEngine (Chromium) 本身占用 200-500MB 内存，启动慢 2-5 秒 |
| **图像传输瓶颈** | 🔴 高 | C++ ↔ JS 图像传输需要序列化（Base64/临时文件），大图像性能差 |
| **GPU 资源竞争** | 🟡 中 | WebEngine GPU 进程与 NCNN Vulkan、Qt RHI 争抢 GPU 资源 |
| **调试复杂** | 🟡 中 | 前端 JS + 后端 C++ + 桥接层，问题定位困难 |
| **打包体积大** | 🟡 中 | WebEngine + Chromium 增加约 100MB |

#### 适用场景评估

| 场景 | WebEngine + React | 原生 Qt/QML |
|------|-------------------|-------------|
| 复杂表单、数据展示 | ✅ 优秀 | 🟡 一般 |
| 图像预览（大图） | ❌ 性能差 | ✅ 优秀 |
| 实时滤镜预览 | ❌ 延迟高 | ✅ 流畅 |
| 视频播放控制 | 🟡 一般 | ✅ 优秀 |
| 开发效率 | ✅ 高（React 生态） | 🟡 中等 |
| 内存占用 | ❌ 高（500MB+） | ✅ 低（100MB） |
| 启动速度 | ❌ 慢（2-5s） | ✅ 快（<1s） |

### 2.2 Shader 模式方案

#### 当前方案
- 基于 Qt RHI / GLSL Shader
- 实时调整：亮度、对比度、饱和度、色相、模糊、锐化

#### ✅ 合理性
- Qt RHI 是正确的选择，支持 Vulkan/D3D11/Metal 跨平台
- GLSL Shader 处理图像效率高

#### ⚠️ 潜在问题
1. **Shader 与 WebEngine 集成困难**
   - WebEngine 内部有自己的 GPU 上下文
   - 需要通过 `QOpenGLWidget` 或 `QQuickWidget` 嵌入，与 WebEngine 窗口共存复杂

2. **实时预览数据流**
   ```
   图像 → C++ Shader 处理 → QImage → Base64/文件 → JS → <img> 显示
   ```
   每帧都需要序列化传输，无法实现真正的实时预览

### 2.3 AI 推理模式方案

#### 当前方案
- NCNN + Vulkan 加速
- Real-ESRGAN 超分辨率模型

#### ✅ 合理性
- NCNN 是成熟的端侧推理框架，无外部依赖
- Vulkan 加速跨平台兼容性好
- Real-ESRGAN 是业界认可的超分辨率模型

#### ⚠️ 潜在问题
1. **模型体积**
   - Real-ESRGAN x4 模型约 17MB
   - Real-ESRGAN x4+ 动漫模型约 17MB
   - 需要考虑模型加载时间和内存占用

2. **处理速度**
   - 1080p 图像 x2 放大约需 0.5-2 秒（取决于 GPU）
   - 4K 视频处理需要逐帧处理，时间较长

3. **与 WebEngine 的内存竞争**
   - WebEngine 占用大量内存
   - NCNN 推理需要额外内存
   - 处理大图像/视频时可能内存不足

### 2.4 视频处理方案

#### 当前方案
- FFmpeg 7.1 硬件编解码
- 逐帧处理

#### ✅ 合理性
- FFmpeg 是行业标准
- 硬件加速正确

#### ⚠️ 潜在问题
1. **视频帧传输**
   - 视频帧需要从 C++ 传到前端显示
   - 大量帧数据传输会成为瓶颈

2. **进度反馈**
   - 视频处理时间长，需要实时进度反馈
   - 通过 QWebChannel 信号频繁更新可能影响性能

## 三、核心问题总结

### 🔴 关键问题：架构选型与功能需求不匹配

**功能需求**：图像/视频画质增强工具
- 大图像预览（4K、8K）
- 实时滤镜预览
- 视频播放控制
- AI 推理处理

**当前架构**：WebEngine + React
- 适合：复杂 UI、数据展示、表单交互
- 不适合：高性能图像处理、实时预览

**根本矛盾**：
1. 图像处理需要高性能数据传输，WebEngine 的 JS 桥接是性能瓶颈
2. 实时滤镜预览需要 GPU 直接渲染，WebEngine 无法直接访问
3. 视频播放需要硬件解码 + GPU 渲染，WebEngine 内部有独立实现，难以与 C++ 处理集成

## 四、改进方案建议

### 方案 A：原生 Qt/QML 架构（推荐）

```
┌─────────────────────────────────────────────────────────────┐
│                    EnhanceVision.exe                        │
│  ┌───────────────────────────────────────────────────────┐  │
│  │              Qt Quick (QML) UI                         │  │
│  │  · 声明式 UI，开发效率接近 React                        │  │
│  │  · 直接访问 C++ 对象，无桥接开销                        │  │
│  │  · GPU 加速渲染（Scene Graph）                         │  │
│  └───────────────────────────────────────────────────────┘  │
│  ┌───────────────────────────────────────────────────────┐  │
│  │              C++ 核心引擎                              │  │
│  │  · AI 推理 (NCNN + Vulkan)                            │  │
│  │  · 图像处理 (Qt RHI Shader)                           │  │
│  │  · 视频处理 (FFmpeg + Qt Multimedia)                  │  │
│  └───────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

#### 优势
| 项目 | 说明 |
|------|------|
| **性能** | 无桥接开销，图像数据直接在 C++ 层处理和显示 |
| **内存** | 减少 300-400MB（无 Chromium） |
| **启动** | < 1 秒 |
| **实时预览** | Shader 可直接渲染到 QML 界面 |
| **视频处理** | Qt Multimedia + FFmpeg 无缝集成 |
| **打包体积** | 减少约 100MB |

#### QML vs React 开发效率
```qml
// QML 示例：图像预览组件
Image {
    id: previewImage
    source: "image://preview/current"  // 直接访问 C++ 图像提供者
    fillMode: Image.PreserveAspectFit
    
    ShaderEffect {
        source: previewImage
        // 实时 Shader 滤镜
        fragmentShader: "qrc:/shaders/brightness.frag"
        property real brightness: slider.value
    }
}
```

```tsx
// React 对比：需要通过桥接获取图像
const [imageUrl, setImageUrl] = useState('');
useEffect(() => {
    bridge.imageProcessor.getCurrentImage().then(url => setImageUrl(url));
}, []);
<img src={imageUrl} />  // 无法直接应用 Shader
```

### 方案 B：混合架构（折中方案）

如果团队坚持使用 React 前端，可采用混合架构：

```
┌─────────────────────────────────────────────────────────────┐
│                    EnhanceVision.exe                        │
│  ┌───────────────────────┐  ┌───────────────────────────┐  │
│  │  Qt 原生窗口          │  │  WebEngine 窗口           │  │
│  │  · 图像预览           │  │  · 设置面板               │  │
│  │  · 视频播放           │  │  · 会话管理               │  │
│  │  · 实时滤镜           │  │  · 文件列表               │  │
│  │  (QML + Shader)       │  │  (React)                  │  │
│  └───────────────────────┘  └───────────────────────────┘  │
│                     ↕ 共享内存 / 信号槽                      │
│  ┌───────────────────────────────────────────────────────┐  │
│  │              C++ 核心引擎                              │  │
│  └───────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

#### 优势
- 保留 React 开发体验
- 图像/视频处理使用原生窗口，性能好
- 复杂 UI 使用 WebEngine，开发效率高

#### 劣势
- 架构复杂，需要管理两个窗口
- 窗口间通信需要额外设计

### 方案 C：优化当前架构（最小改动）

如果必须使用 WebEngine + React，需要优化：

#### 1. 图像传输优化
```cpp
// 使用共享内存传输图像
class SharedImageProvider : public QQuickImageProvider {
public:
    QImage requestImage(const QString& id, QSize* size, const QSize& requestedSize) override {
        // 直接从共享内存读取，无需序列化
        return m_sharedMemory.readImage(id);
    }
};
```

#### 2. 实时预览优化
- 使用 WebCodecs API（Chrome 94+）进行视频解码
- 使用 OffscreenCanvas + WebGL 进行图像处理
- 但这会与 C++ 后端处理逻辑分离

#### 3. 进度反馈优化
```typescript
// 使用节流减少桥接调用
const throttledProgress = throttle((progress) => {
    useProcessingStore.getState().setProgress(progress);
}, 100);  // 每 100ms 最多更新一次
```

## 七、结论

### 当前方案的主要问题
1. **架构选型与功能需求不匹配**：WebEngine + React 不适合高性能图像处理应用
2. **性能瓶颈**：图像传输、实时预览、内存占用都存在严重问题
3. **开发复杂度高**：多层抽象增加调试难度

### 推荐方案
**方案 A：原生 Qt/QML 架构**
- 性能最优，开发效率可接受
- QML 声明式语法与 React 类似，学习成本低
- 完美支持 Shader、视频、AI 推理集成

### 如果坚持 WebEngine
需要采用**方案 B：混合架构**或**方案 C：深度优化**
- 接受性能折衷
- 增加开发复杂度
- 需要更多优化工作
